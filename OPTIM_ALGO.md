# 装备优化算法说明

## 1. 要解决的问题

这个功能要解决的是一个配装搜索问题：给定一个角色职业、等级、当前页面里可选的装备范围、可选食品范围，以及已经固定的装备和魔晶石，自动找出一套伤害期望最高且满足 GCD 约束的方案。

一套完整战斗配装通常包含这些选择：

- 武器或主副手。
- 头、身、手、腿、脚。
- 耳、项链、手镯。
- 两个戒指位。
- 食品。
- 每件装备上的若干魔晶石。

装备本身提供主属性、暴击、直击、信念、咏速/技速、坚韧、信仰、武器性能等属性。魔晶石可以给装备补充副属性，但受装备上限限制；食品则通常给 2 到 3 个属性加成，且每项都有百分比上限。

优化目标当前是“每威力伤害期望最大”。也就是把一套配装的最终属性代入战斗伤害公式，计算一次标准化后的期望伤害，然后寻找数值最高的方案。未来目标可能扩展成用户可配置公式，例如同时考虑伤害、GCD、减伤率、MP 回复等，但当前实现里 GCD 是约束，不是优化目标。

这个问题不是简单地“每个部位选属性最高的装备”。原因包括：

- 暴击、直击、信念等属性通过带取整的公式影响伤害，单个属性点的边际收益不固定。
- 魔晶石有装备属性上限，同一颗魔晶石在不同装备上可能有效、部分浪费或完全不可用。
- 食品按当前属性百分比加成，某些裸属性较低的方案吃食品后可能反超。
- GCD 由咏速/技速决定，必须落在指定范围内，但咏速/技速本身不一定越高越好。
- 两个戒指位不能使用同一个物品。

因此，这本质上是带约束的组合优化问题。

## 2. 搜索空间数量级

### 2.1 装备选择

如果每个非食品部位平均有 `k` 件候选装备，战斗职业大约有 11 到 12 个装备位。那么只考虑装备、不考虑魔晶石和食品，组合数大约是：

```text
k^12
```

几个直观数量级：

- 每部位 2 件候选：约 4,096 种装备组合。
- 每部位 3 件候选：约 531,441 种装备组合。
- 每部位 5 件候选：约 244,140,625 种装备组合。

实际页面会受筛选范围和已固定装备影响，数量可能小很多。但只要有多个未固定部位，暴力枚举完整装备组合就会很快不可接受。

### 2.2 魔晶石选择

魔晶石是更大的问题。假设一个空孔可选 `m` 种属性，一件装备有 `s` 个空孔，则该装备的魔晶石局部组合数约为：

```text
m^s
```

在当前战斗职业场景中，常见可选副属性大约是 4 到 6 种。即使当前已经做了“只考虑最高等级、溢出则不考虑该属性、只枚举镶满方案”的收缩，单件装备仍可能有不少组合：

- 2 孔、4 种属性：16 种。
- 3 孔、4 种属性：64 种。
- 5 孔、4 种属性：1,024 种。

如果装备全部固定，只优化魔晶石，理论组合数仍然可能很高。假设 10 件装备各有 2 个空孔、每孔 4 种可选属性，则总组合数是：

```text
4^20 = 1,099,511,627,776
```

这解释了为什么“装备已经选好，只挑魔晶石”仍然可能跑不动。装备固定只消除了装备维度，没有消除魔晶石的指数增长。

### 2.3 食品选择

食品数量相对小，通常是十几个到几十个量级。当前实现把食品放在搜索最后枚举，所以食品不是状态爆炸的主因。但食品会影响最终分数，也会让中间剪枝更难安全判断。

## 3. 约束和保留条件

当前优化器遵循这些约束：

- 只优化战斗职业。
- 已经选择的装备固定不动。
- 已经选择的食品固定不动。
- 已经镶嵌的魔晶石固定不动。
- 没有选择的装备部位，只从当前页面筛选结果中取候选。
- 聚焦模式下，只考虑聚焦模式过滤后的候选。
- 没有选择食品时，只从默认推荐的最优食品范围中取候选。
- 同一个物品只能使用一次，主要影响两个戒指位。
- 魔晶石只填空孔，不改动已有魔晶石。
- 当前只枚举镶嵌满的魔晶石方案。
- 每个空孔只考虑可镶嵌的最高数值魔晶石。
- 如果某颗魔晶石会使该装备对应属性溢出，则该孔不考虑该魔晶石。
- GCD 必须满足用户配置的下限、上限或精确目标值。

这些约束有两个目标：一是保证结果符合用户当前编辑上下文；二是尽量提前缩小搜索空间。

## 4. 当前算法概览

当前算法不是一次性暴力枚举完整配装，而是按装备位逐步合并状态，过程中不断剪枝。可以理解为动态规划加 Pareto frontier。

整体流程：

1. 计算可接受的咏速/技速范围。
2. 为每件候选装备生成所有可能的局部方案。
3. 对每件装备的局部方案做一次局部剪枝。
4. 按装备位顺序，把上一轮状态和当前部位局部方案合并。
5. 每合并一个部位后，按精确 speed 分桶。
6. 每个 speed 桶内做目标阈值剪枝和 Pareto 剪枝。
7. 所有装备位处理完成后，再枚举食品。
8. 计算最终伤害，过滤不满足 GCD 的结果，排序输出。

搜索已经放在 Web Worker 中执行，所以不会阻塞浏览器主线程。主线程只负责启动、取消、展示进度和应用结果。

## 5. 局部方案

局部方案是“某件装备加上它的可变魔晶石选择”之后得到的小方案。它包含这件装备最终贡献的属性、speed、装等贡献、使用的魔晶石，以及这个方案是否改变了当前配装。

举例来说，一件有 2 个空孔的装备，每个孔可以选暴击、直击、信念、技速。如果都不溢出，那么它会生成 16 个局部方案：

```text
暴击/暴击
暴击/直击
暴击/信念
暴击/技速
...
技速/技速
```

局部方案生成后，会先在同一件装备内部做支配剪枝。若两个局部方案 speed 相同，并且方案 A 在所有收益属性上都不低于方案 B，且至少一个属性更高，那么 B 不可能优于 A，可以删除。

这一步能减少一些明显无效的魔晶石组合，但不能解决全局状态爆炸，因为很多方案互有优劣。例如一个方案暴击更高，另一个方案信念更高，它们都不能直接删。

## 6. 全局状态

全局状态表示“已经处理完前若干部位后，当前累计出来的一种部分配装”。

一个状态至少需要记录：

- 当前累计属性，不含食品。
- 当前累计咏速/技速。
- 已经选择的装备路径。
- 已经使用过的物品，用于防止重复戒指。
- 当前平均装等计算所需信息。
- 当前改动数量和魔晶石数量。
- 当前目标函数分数缓存。

早期实现中，每个状态都保存对象、映射和集合。状态数增长后，会产生大量对象复制和垃圾回收。当前实现已经改成数组池：状态只保存数字索引、定长属性数组、前驱状态索引和本轮选择的局部方案索引。最终需要输出结果时，才沿前驱链回溯出完整装备和魔晶石方案。

这个改动降低了常数开销，但没有改变搜索空间本身。

## 7. GCD 和 speed 的处理

GCD 由咏速或技速计算。当前算法先反推出满足用户 GCD 配置的 speed 范围，然后在搜索中做可行性剪枝：

- 如果当前 speed 已经超过允许最大值，丢弃。
- 如果当前 speed 加上剩余部位最多能提供的 speed 仍低于允许最小值，丢弃。

speed 不能简单作为普通收益属性处理，也不能粗暴分桶。

例如目标需要至少 1000 speed。当前两个部分状态分别是 900 和 940。它们可能处在同一个 GCD 档附近，但如果后续有一件装备提供 100 speed，那么 900 会刚好到 1000，940 会到 1040，可能浪费 40 点 speed。反过来，如果后续只能提供 60 speed，那么 940 能达标，900 不能达标。

所以不同 speed 的状态不是完全等价的。当前实现按精确 speed 分桶，只在同 speed 内比较和剪枝。这更保守，但也让跨 speed 的剪枝能力很弱。

## 8. 每轮剪枝

每处理一个装备位后，会产生一批新状态。当前剪枝分两层。

第一层是目标阈值剪枝。对同一个 speed 桶内的状态计算当前伤害期望分数。如果配置了剪枝阈值，例如 99%，则只保留不低于该 speed 桶最优分数 99% 的状态。

这是启发式剪枝，不是严格正确剪枝。它能快速减少状态数，但可能误删后续通过食品或后续装备反超的状态。

第二层是 Pareto frontier。仍然只在同一个 speed 桶内比较。若状态 A 在所有收益属性上都不差于状态 B，并且至少一个属性更好，则 B 被支配，可以删除。如果收益属性完全相同，则改动更少、魔晶石更少的状态优先。

这里的收益属性不包含 speed，因为 speed 已经作为分桶条件。典型收益属性包括主属性、武器性能、暴击、直击、信念、坚韧等，具体取决于职业。

Pareto frontier 的优点是相对安全：被支配的状态理论上没有保留价值。缺点是复杂度高。如果一个 speed 桶内有 N 个互不支配的状态，就需要大量两两比较，最坏接近 O(N^2)。

## 9. 食品处理

食品在装备和魔晶石搜索完成后才枚举。这样能避免把食品也并入中间状态，减少状态维度。

但这个设计有一个副作用：搜索过程中的目标阈值剪枝使用的是“不含食品”的分数，而最终排序使用的是“含食品”的分数。某个状态裸分较低，但由于它的属性更适合某个食品，最终可能变得更好。当前阈值剪枝可能提前删掉这种状态。

因此，食品后置是一种性能和准确性的折中。

## 10. 当前主要瓶颈

### 10.1 魔晶石组合指数增长

即使装备固定，空魔晶石孔仍然会形成指数级组合。每多一个空孔，状态数大约乘以该孔可选属性数。当前“只枚举镶满方案”减少了空孔可跳过的分支，但仍然保留了所有属性选择分支。

### 10.2 Pareto 比较成本高

状态数只有几千时仍然慢，主要原因通常不是几千本身，而是几千状态之间的支配比较。

如果一个 speed 桶中有 5,000 个状态，并且它们大多互不支配，frontier 构建最坏可能需要约 25,000,000 次状态比较。每次比较又要检查多个属性维度。再乘以多个 speed 桶和多个装备位，耗时会很明显。

### 10.3 speed 限制削弱跨桶剪枝

因为 speed 不能简单合并，很多属性上明显相近的状态只要 speed 不同，就不会互相支配。这保留了正确性，但让状态更多地留存下来。

### 10.4 目标函数存在取整阈值

FFXIV 伤害公式中有大量取整。属性多一点不一定提高伤害，少一点也不一定降低伤害。简单的线性评分或属性加权上界不够可靠，导致很难设计既强又安全的剪枝。

### 10.5 JS 数据结构只解决常数问题

数组池减少了对象分配和 GC，能提升热路径性能。但如果算法层面仍然要做千万级甚至更高数量级的支配比较，语言和数据结构优化只能缓解，不能根治。

迁移到 Rust 或 C# 可能降低常数开销，尤其是多维比较、数组访问和多线程方面。但如果不改变状态数量和比较复杂度，提升会有上限。真正的关键仍然是减少需要保留和比较的状态。

## 11. 后续优化方向

可以考虑以下方向：

- 为“装备固定，只优化魔晶石”设计专用算法。先对每件装备内部生成更小的 frontier，再合并装备 frontier。
- 为 Pareto frontier 建索引结构，避免每个新状态都和所有 frontier 状态比较。
- 对属性向量做整数编码或压缩，降低单次比较成本。
- 增加详细 profiling：每个部位的局部方案数、合并前状态数、合并后状态数、阈值剪枝后状态数、frontier 后状态数和耗时。
- 引入可选近似模式，例如 beam search，但必须在 UI 上说明它可能错过全局最优。
- 研究食品上界或提前纳入食品，减少阈值剪枝误删最终高分状态的风险。

## 12. 改进记录

这里记录已经落地的优化改进。每个改进点尽量说明它解决的问题、采用的做法，以及对搜索语义和性能的影响。

### 12.1 魔晶石局部方案按最终属性向量去重

原实现逐孔递归枚举魔晶石分配。若多个空孔拥有相同可选属性和相同魔晶石等级，很多排列会得到相同的最终装备属性。例如 5 个孔都可选暴击、直击、信念和技速时，暴击 2 颗、信念 3 颗的不同孔位排列对最终属性没有区别，但会作为不同局部方案进入后续搜索。

改进后，局部方案生成改为小型 DP：

```text
frontier = { 装备裸属性 + 已固定魔晶石 }

for each empty materia slot:
    next = {}
    for state in frontier:
        for option in slot.options:
            newStats = applyMateriaWithCap(state, option)
            key = finalStatsVector(newStats)
            next[key] = bestRepresentativeAssignment(key)
    frontier = next
```

也就是说，同一件装备内部按“最终属性向量”合并等价方案，只保留一个可回填到 UI 的代表魔晶石分配。这样可以在进入全局状态合并前删除排列型重复，尤其能改善“装备固定，只优化魔晶石”的场景。

该改进不改变属性封顶规则：每次应用魔晶石仍按装备裸属性和装备上限计算有效属性，溢出的部分不会进入最终属性。

### 12.2 两个戒指槽合成组合槽

原实现把两个戒指槽作为两个独立阶段处理，并在热路径中沿状态前驱链检查 `itemId`，避免同一个戒指物品被使用两次。这个约束带有路径依赖：处理第一个戒指后，状态未来可选集合取决于已经使用了哪个物品。

改进后，优化器在局部方案阶段把两个戒指槽合成为一个组合槽：

```text
ringPairPlans = {
    leftRingPlan + rightRingPlan
    | leftRing.itemId != rightRing.itemId
}
```

合并后的戒指对作为一个普通候选部位进入全局搜索。这样可以提前排除重复戒指组合，减少全局搜索阶段的路径依赖。最终输出结果时，组合槽会拆回原来的两个戒指槽位，因此 UI 应用结果的格式保持不变。

戒指组合生成后还会执行一次局部 Pareto 剪枝，删除同 speed 下被收益属性支配的戒指对，避免明显无效的组合继续进入全局 DP。

### 12.3 同 speed 桶 Pareto 改为离线 skyline

原 `buildParetoFrontier` 是在线插入式 frontier：每个新状态先检查是否被已有 frontier 支配，再反向删除被它支配的旧状态。桶内状态较多且大多互不支配时，复杂度接近：

```text
O(N^2 * d)
```

改进后，桶内状态在进入 skyline 前先做 canonicalize：

```text
相同收益属性向量 -> 只保留改动数、魔晶石数更优的代表状态
```

之后删除桶内常量收益维度，只对实际发生变化的有效维度构建 skyline。小桶仍使用朴素算法，避免索引结构的常数成本；较大桶使用离线 skyline：

- 1 维：只保留最大值。
- 2 维：排序后线性扫描。
- 3 维及以上：排序后用稀疏 Fenwick tree 维护后续维度的最大值。

这将常见低维桶的剪枝从平方级比较降为排序和索引查询，直接缓解单个 speed 桶状态数达到几千时的卡顿问题。

需要注意的是，精确 speed 分桶策略仍然保留。该改进没有做跨 speed 支配，也没有改变目标阈值剪枝的启发式性质。
